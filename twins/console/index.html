Sales Twin Console

The approval surface for the compliance-native sales twin. Every draft, reply, and guardrail verdict lands here for a human to approve, hold, or edit before anything is ever sent. This build ports the real decision logic from the twin's Python guardrails so the sandbox and scorecard below actually compute.

Demo · runs entirely in your browser Seeded sample data. No account, no network calls, nothing sent anywhere. All guardrail logic and scoring run client-side in this page. Production sending needs a backend and keys (see the notes on each tab).

Pipeline

One state machine across email and LinkedIn. Contacts advance researched → drafted → awaiting-approval → booked. Fit score gates who is contacted at all; a low fit is suppressed and logged, never sent.

Review queue human-in-the-loop

Nothing reaches a delivery adapter without clearing all three rails and a human approval. For each drafted message you see the draft, its fit score, and the three guardrail verdicts computed live: firewall (clean / redacted / blocked), QA (respond / hold / veto), and hard-rules (pass / blocked).

Production hook: Approve wires to the deterministic delivery adapter (email with SPF/DKIM/DMARC, or a human-confirmed LinkedIn action). Edit reopens the draft for a human rewrite that re-runs the rails. Here Approve/Hold/Edit update local demo state only.

Guardrail sandbox

Paste an inbound reply and screen it. This runs the ported injection firewall (regex + homoglyph-fold + base64 decode + payload-split detection) and the QA reply-worthiness veto, the same logic as the twin's Python rails. The verdict tells you whether the twin would respond, stay silent on purpose, or quarantine an injection to the console.

Production hook: the deterministic screen shown here always runs first, so safety never depends on a model. In production an LLM classifier layers on as an optional second opinion behind the same seam. No LLM key is used in this demo.

This is a demonstration console.

It runs entirely client-side with seeded sample data and sends nothing anywhere. A production deployment requires a backend and API keys: an LLM provider for drafting and classification, an email sending domain with SPF, DKIM, and DMARC aligned and warmed, human-approved LinkedIn actions (no automation or scraping, per LinkedIn User Agreement §8.2), an append-only audit log, and the Trust-Safe governance layer (consent, provenance, disclosure, substantiation, brand-voice, guardrails).
